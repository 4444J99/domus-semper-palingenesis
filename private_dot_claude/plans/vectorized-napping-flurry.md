# Plan: The System Library — Prompt Chains, Plan Catalog, SOP Lifecycle

## Context

This session collected 5 layers of prompt grammar, 12 operational vocabulary categories, 22 frameworks, and 7 paired patterns from 14+ sessions across 9 repos. All currently stored as memory files — per-conversation ephemera that vanish when context compresses. Meanwhile: 269 plan files across 49 repos, 73 SOPs in praxis-perpetua, 37 memory directories — all dispersed, undiscoverable by agents in other sessions.

The user's directive: prompts become sequenced prompt chains. Plans stored in the system's library. SOPs discovered through rep → absorb → fortify → place → chain. Universal directories that every project references.

**The problem:** Agents see fragments, not the whole. The solution is a permanent library that any session discovers through 4 lines in its CLAUDE.md.

## Architecture

**Location:** `meta-organvm/praxis-perpetua/library/` — extends the existing SOP corpus (GRADUATED, 67 standards). The library is process governance data, not code.

```
praxis-perpetua/library/
├── plans/
│   ├── index.json          # Machine-readable catalog (generated by organvm plans catalog)
│   ├── INDEX.md            # Human-readable catalog (generated)
│   └── by-domain/          # YAML pointer files per domain (governance.yaml, research.yaml, ...)
├── chains/
│   ├── session-protocol.yaml    # 8 phases: SEED→HANDOFF
│   ├── design-grammar.yaml      # 9 phases: DEFINE→TRANSFORM
│   ├── close-protocol.yaml      # 12 steps, all mandatory
│   ├── sop-lifecycle.yaml       # rep→absorb→fortify→place→chain
│   ├── ingest-execute.yaml      # Paired pattern chain
│   ├── audit-close.yaml         # Paired pattern chain
│   ├── auditor-pair.yaml        # Above (fleet gap scan) + Below (reverse lookup disposition)
│   └── INDEX.md
├── vocabulary/
│   ├── vocabulary.yaml          # 12 categories with trigger phrases
│   ├── paired-patterns.yaml     # 7 paired patterns
│   ├── frameworks.yaml          # 22+ frameworks & principles
│   └── INDEX.md
└── directory-canon.yaml         # Universal/common/situational directory spec
```

## What Migrates from Memory → Library

| Memory File | Library Destination | Format Change |
|-------------|-------------------|---------------|
| `project_prompt_sequence_protocol.md` | `chains/session-protocol.yaml` | Narrative → executable YAML chain |
| `project_design_grammar.md` | `chains/design-grammar.yaml` | Narrative → executable YAML chain |
| `feedback_close_protocol.md` | `chains/close-protocol.yaml` | Narrative → executable YAML chain |
| `project_operational_vocabulary.md` | `vocabulary/vocabulary.yaml` | Markdown → structured YAML dispatch table |
| `project_frameworks_and_principles.md` | `vocabulary/frameworks.yaml` + `directory-canon.yaml` | Split: frameworks to YAML, canon to its own file |

Memory files become historical references, not active sources. New sessions read from the library.

## Step 2: Extract Chains from Memory

**Files:** `praxis-perpetua/library/chains/*.yaml`

Convert each grammar layer from narrative markdown to YAML using the existing `skill-chain-prompts` format:

```yaml
# Example: session-protocol.yaml
name: session-protocol
description: 8-phase session steering grammar
steps:
  - id: SEED
    description: Single intent-laden sentence before context arrives
    recognition: ["providing info for", "review:", bare intent sentence]
    behavior: Absorb, don't act
    checkpoint: false
  - id: REDIRECT
    description: Calibration correction after first output
    recognition: ["the report should stop at", "that's your translation"]
    behavior: Tighten, don't defend
    checkpoint: false
  # ... 6 more steps
```

Each chain file: ~50-80 lines. Recognition criteria from memory files. Behavior directives as actionable instructions.

## Step 3: Extract Vocabulary to YAML

**Files:** `praxis-perpetua/library/vocabulary/*.yaml`

Convert operational vocabulary from markdown categories to structured dispatch:

```yaml
# Example entry in vocabulary.yaml
categories:
  - name: INGEST
    trigger_phrases:
      - "full 100% ingest"
      - "100% contextual awareness"
      - "full dissection"
    behavior: "Read EVERYTHING before acting. No skimming. No assumptions."
    pairs_with: EXECUTE
```

Frameworks as structured entries with invocation order metadata.

## Step 4: Plan Catalog Engine

**Files:** `organvm-engine/src/organvm_engine/plans/index.py`, `organvm-engine/src/organvm_engine/cli/plans.py`

1. Add `domain` field to `PlanEntry` — classified from tags using `distill/taxonomy.py`
2. New CLI: `organvm plans catalog --write` — builds index, writes to `library/plans/index.json` + `INDEX.md` + `by-domain/*.yaml`
3. New CLI: `organvm plans search <query>` — searches committed index by title/tags/domain/organ

Plans stay where they are (`.claude/plans/` in each project). The library indexes them with pointers.

## Step 5: SOP Lifecycle Field

## What This Does NOT Do

- Does NOT replace memory files (memory = session context, library = permanent canon)
- Does NOT create a database (everything is markdown + YAML + JSON in git)
- Does NOT auto-execute chains (agents follow chains, the library declares them)
- Does NOT centralize plan files (plans stay project-local, the library indexes them)
- Does NOT require new repos (extends praxis-perpetua + organvm-engine)
