## Where notebooks fit (under the recursive lens)

A notebook is a **first-class recursive instrument**: it executes, observes its execution, and modifies its own behavior based on observations (cells re-running based on prior cell outputs). Per skill categorization:

- **Direct self-reference**: a notebook cell can reference and modify earlier cells via the IPython kernel
- **Hierarchical**: notebook header (Lane / Status / Inputs / Outputs) is meta-data ABOUT the notebook contents
- **Observer-observed**: notebooks observing the substrate ARE part of the substrate ‚Äî the strange loop to design carefully
- **Recursive decomposition** (skill ¬ßRecursive Decomposition): a notebook breaking analysis into cells, each cell-output feeding the next, IS recursive decomposition

**Practical implication for the original 7-lane plan**: Lane 1 (Corpus EDA) is the most recursively-loaded lane because it studies the substrate that produced the notebook studying it. This calls for **explicit observer-observed discipline**: the notebook must declare in its header that it's analyzing data which includes its own creation event. Without this discipline, repeated EDA notebooks introduce cumulative observer effects that distort future analyses.

**New mandatory header field for Lane 1 notebooks**:
```yaml
observer_effect:
  observes_data_that_includes_self: true|false
  if_true_isolation_strategy: "snapshot at T-1; analyze snapshot only; do not re-read live data after notebook creation"
```
