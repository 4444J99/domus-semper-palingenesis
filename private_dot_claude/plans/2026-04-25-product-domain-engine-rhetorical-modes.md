# Product Domain Engine — System of Formalizations
## The meta-system that generates Hokage Chess, Styx, Elevate Align, and all future product domains

## Context

Every product tied to a domain is an instance of the same operation: take a domain that exists as informal knowledge, lived experience, and scattered practice — and **formalize it into computable structure** that derives products, content, and community as outputs.

This isn't a framework you "apply." It's a recognition that what already happened across four products (public-record-data-scrapper, styx, elevate-align, hokage-chess) IS a system — one that was never named, never encoded, and therefore re-derived every time. This plan formalizes the engine itself.

The addition of **rhetorical modes** (logos, ethos, pathos) adds a second axis: every formalization expresses through three independent registers. The domain formalization is the vertical axis (what is true). The rhetorical mode is the horizontal axis (how truth faces outward). Every product output is a coordinate in this space.

## Part 1: The Formalization Protocol

Every domain gets the same five-phase treatment:

### Phase 1: IDENTIFY — The Domain's Agents
Who acts in this space? What are their roles, incentives, constraints?

| Domain | Agents |
|--------|--------|
| Chess improvement | improving player, opponent, coach, platform algorithm, community |
| Public records | filer, jurisdiction, searcher, aggregator, regulator |
| Behavioral economics | individual, peer auditor, stake mechanism, ledger, market |
| Health/wellness | practitioner, client, protocol, cycle, measurement system |

### Phase 2: MAP — The Domain's Structures
What are the real relationships, hierarchies, flows, and constraints?

| Domain | Structures |
|--------|-----------|
| Chess | ELO bands → skill gaps → study methods → rating movement → plateau cycles |
| Public records | state → filing office → document type → access method → data schema |
| Styx | behavior → commitment → stake → peer audit → accountability feedback loop |
| Wellness | symptom → protocol → cycle → measurement → adjustment → practitioner guidance |

### Phase 3: ENCODE — Structures as Computable Objects
Not concepts, not documents — types, functions, tests. The formalization becomes executable.

| Domain | Computable Objects |
|--------|-------------------|
| Chess | `VideoMetrics`, `NarrativeStructure`, `GrowthTarget`, `ThumbnailScore`, `scoreTitleFormula()` |

### Phase 4: EXPRESS — Through Three Rhetorical Modes
Each formalization speaks through three independent channels. These are not "communication styles" — they are **independent output types** from the same underlying truth.

### Phase 5: DEPLOY — Let Density Create Gravity
The public-record-data-scrapper principle: if the formalization is dense enough and real enough, it attracts its own audience. The 4,455-word README, the 60+ collection agents, the 2,055 tests — that mass created gravitational pull. Organic traffic from existence, not promotion.

## Part 2: The Rhetorical Modes as Independent Output Channels

### LOGOS (λόγος) — The Internal Skeleton
**What is structurally true.** The formalization itself. Logos is the INTERNAL architecture — the types, proofs, algorithms, schemas that make the system coherent. Logos rarely faces the customer directly. Its function is to make ethos and pathos POSSIBLE. Without logos, ethos is hollow and pathos is manipulation.

Logos is what you build. It's not what you show.

Outputs (internal):
- Type definitions and interfaces
- Algorithm implementations
- Data schemas and pipelines
- Test suites (internal validation)
- Financial models
- System architecture

### ETHOS (ἦθος) — The Credibility/Authority Register (Built Through Academic Density)
**Why it should be trusted.** Ethos is generated by PRODUCING scholarly-grade research, formal proofs, and quantifiable evidence. The audience doesn't read the proofs — they see that proofs EXIST and trust the source.

**The mechanism**: Academic studies, research papers, test counts, formal citations, peer-reviewed source lists, deployment metrics. These are logos internally but their FUNCTION facing outward is ethos. "2,055 tests" is not information for the customer — it's a credibility signal. "9 theorem proofs" is not education — it's authority.

Outputs (external-facing):
- Academic studies and research papers → signals depth
- Test counts and CI badges → signals reliability
- Portfolio README with provenance → signals professionalism
- Peer-reviewed source lists (30-80 sources per pillar) → signals rigor
- Deployment URLs and uptime metrics → signals operational reality
- Case studies with measurable outcomes → signals results
- Trademark filings and legal standing → signals legitimacy
- PROVENANCE.yaml with SHA-256 hashes → signals integrity

**Generation method**: $RESEARCH_ATLAS, dissertation-grade analysis, canonical paper lists, experiment design packs. The DEPTH of the research is the raw material that ethos is forged from.

Audience: sponsors, employers, investors, potential clients, collaborators, the market.

### PATHOS (πάθος) — The Emotional/Narrative Register (Built Through Social & Media Content)
**Why it matters to a human.** Pathos is generated by PRODUCING social and media content that creates emotional connection, belonging, and identity. The audience doesn't analyze the narrative structure — they FEEL it.

**The mechanism**: YouTube videos, Discord community, email sequences, landing pages, brand voice, naming systems, physical objects. These create the feeling of "I'm part of this" and "this is for me."

Outputs (external-facing):
- YouTube content (the evergreen library) → creates connection
- Shorts / Reels / TikTok → creates discovery
- Landing pages (conversion-optimized) → creates desire
- Pitch decks → creates vision
- Discord community → creates belonging
- Email sequences → creates relationship
- Brand voice and visual identity → creates recognition
- Naming systems (Genin → Hokage) → creates identity
- Physical objects (chess pieces, merch) → creates tangibility
- The underdog / origin story → creates emotional investment

**Generation method**: Content strategy, narrative frameworks (Ki-Shō-Ten-Ketsu), hook architecture, community design. The VOLUME and AUTHENTICITY of the media is the raw material that pathos is forged from.

Audience: customers, community members, followers, the human who encounters the product.

### The Relationship Between Modes

```
LOGOS (internal skeleton)
  │
  ├──→ ETHOS (academic density makes credibility visible)
  │     "We have 2,055 tests" = logos fact, ethos function
  │     "9 peer-reviewed proofs" = logos fact, ethos function
  │
  └──→ PATHOS (narrative makes emotional connection)
        "I'm 1350 and climbing" = logos fact (his actual rating), pathos function
        "322 documented games" = logos fact (real count), pathos function (commitment signal)
```

Logos is never customer-facing on its own. It ALWAYS expresses through ethos or pathos. The customer sees credibility (ethos) or feels connection (pathos). They never see the type definitions.

## Part 3: The Composition Matrix

Every product output is a specific BLEND of the three modes. The blend ratio determines what the output IS.

| Output Type | Internal (Logos) | External Function | Generation Method |
|-------------|-----------------|-------------------|-------------------|
| Technical spec | ■■■ | — (internal only) | Engineering |
| Academic study | ■■■ internally | **ETHOS** externally | $RESEARCH_ATLAS, canonical papers |
| Test suite | ■■■ internally | **ETHOS** externally ("2,055 tests") | TDD, CI/CD |
| README | ■■ internally | **ETHOS** externally | Documentation craft |
| Financial model | ■■■ internally | **ETHOS** externally | Unit economics research |
| Source list | ■■■ internally | **ETHOS** externally ("80 peer-reviewed sources") | Scholarly bibliography |
| Landing page | ■ internally | **PATHOS** externally | Content strategy, conversion design |
| YouTube content | ■ internally | **PATHOS** externally | Narrative frameworks, hook architecture |
| Community | □ internally | **PATHOS** externally | Discord design, identity systems |
| Pitch deck | ■ internally | **PATHOS + ETHOS** | Narrative + evidence |
| Course content | ■■ internally | **PATHOS + ETHOS** | Educational design + credibility |
| Physical product | □ internally | **PATHOS** externally | Brand embodiment |
| Deployment URL | ■■ internally | **ETHOS** externally ("it's live") | DevOps |

**Key insight**: Every artifact has a logos core (internal truth) but its FUNCTION for the audience is always ethos (trust) or pathos (feeling) or both. The customer never consumes logos directly. They consume the credibility or emotion that logos enables.

**The two generation engines**:
1. **Academic density → ETHOS**: Research papers, source lists, test suites, proofs, deployment metrics. The MORE rigorous and dense the scholarly work, the stronger the ethos signal.
2. **Social/media content → PATHOS**: YouTube, Discord, email, landing pages, brand voice, physical objects. The MORE authentic and voluminous the media, the stronger the pathos signal.

## Part 5: The Prompt Sequence (Universal Build Commands, Correctly Ordered)

The user's prompt atoms, resequenced for the Product Domain Engine:

### Tier 1: Foundation (what is this domain?)
```
$MODE = "lean MVP (fast deployment)" | "full-stack system (multi-service, production-grade)"
$DEPTH = "operator-grade" | "dissertation-grade"
Formalize Natural Center as a computable object (not a concept)
Constraint-domain pillars (what can break or limit the system in reality)
```

### Tier 2: Research (scholarly study of the domain)
```
$RESEARCH_ATLAS_V3
or a full dissertation-grade atlas with milestones + outputs
exact peer-reviewed source list per pillar (30–80 sources)
Construct the exact reading ladder + source list (peer-reviewed, canonical)
exact canonical papers per pillar
Methodological rigor pillars (how you prove things are true)
$PILLAR_CAUSAL_INFERENCE
$PILLAR_UNIT_ECONOMICS
$PILLAR_ALGORITHMIC_INTERFACE
```

### Tier 3: Architecture (how does it become a system?)
```
Translate this into a full repository architecture + file tree + environment variables
$REPO_ARCHITECTURE_CME_FULL
build full repo architecture ($ENV variables, services, pipelines)
define brand embedding structure
formalise Natural Center extraction algorithm
Define exact algorithms for clip extraction and scoring
Build the attribution model mathematically
```

### Tier 4: Product Expression (logos → ethos → pathos outputs)
```
produce a sales one-pager vs technical spec split          [LOGOS + ETHOS]
create a landing page version (conversion-optimized)        [PATHOS + ETHOS]
tighten this into a pitch deck narrative (10–12 slides)     [PATHOS + ETHOS]
$NEXT = "EXPERIMENT DESIGN PACK (ready-to-run tests)"      [LOGOS]
a live research operating system (ROS) spec                 [LOGOS]
full experiment design templates                            [LOGOS]
```

### Tier 5: Deployment & Scaling
```
$NEXT = "SERVICE-BY-SERVICE CODE SCAFFOLD + FIRST DEPLOYMENT SCRIPTS"
$NEXT_ARTIFACT = "FULL_STACK_IMPLEMENTATION_BLUEPRINT"
expand into a multi-tenant SaaS system with orchestration layer
$PROC_NATURAL_CENTER_BOOTSTRAP
```

**Order per instance depends on where the domain is:**
- New domain (hokage-chess): Tier 1 → 2 (light) → 3 → 4 → 5
- Theory-heavy domain (styx): Tier 1 → 2 (deep) → 3 → 4 → 5
- Deployed domain (public-record-data-scrapper): Tier 3 → 4 → 5 (tiers 1-2 already done)
- Client domain (elevate-align): Tier 1 → 4 → 3 → 5 (pathos first, architecture follows)

```

├── src/                    # LOGOS — the formalization
│   ├── lib/                # Domain types, functions, algorithms
│   └── app/                # Product surface (landing page, app, API)
├── research/               # LOGOS — scholarly study
│   ├── pillars/            # One file per research pillar
│   ├── sources/            # Canonical papers and references
│   └── atlas.md            # Research atlas with milestones
├── reference/              # ETHOS — provenance and evidence
│   ├── PROVENANCE.yaml     # SHA-256 chain of source materials
│   ├── case-studies/       # Measurable outcomes
│   └── competitive/        # Market analysis, gap analysis
├── brand/                  # PATHOS — narrative and identity
│   ├── voice.md            # Brand voice definition
│   ├── naming.md           # Naming system (like Genin→Hokage)
│   ├── visual.md           # Color, typography, imagery
│   └── narrative.md        # Origin story, positioning angle
├── outputs/                # Composed outputs (mode blends)
│   ├── landing-page/       # PATHOS + ETHOS
│   ├── pitch-deck/         # PATHOS + ETHOS
│   ├── technical-spec/     # LOGOS
│   ├── one-pager/          # ETHOS + PATHOS
│   └── financial-model/    # LOGOS + ETHOS
├── tests/                  # LOGOS + ETHOS (proof + credibility)
├── seed.yaml               # ORGANVM contract
└── CLAUDE.md               # Agent instructions
```

### Artifact 3: Rhetorical Mode Audit Function
TypeScript function that scores any repository against the three modes:

```typescript
interface RhetoricalAudit {
  domain: string;
  logos: { score: number; evidence: string[]; gaps: string[] };
  ethos: { score: number; evidence: string[]; gaps: string[] };
  pathos: { score: number; evidence: string[]; gaps: string[] };
  dominant_mode: "logos" | "ethos" | "pathos";
  composition_gaps: string[];  // outputs that should exist but don't
}
```

Scans for: test files (logos), README quality (ethos), landing page existence (pathos), type definitions (logos), social proof data (ethos), brand assets (pathos), research documents (logos), deployment status (ethos), narrative content (pathos).
