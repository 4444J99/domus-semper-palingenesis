How any one thing is done = how every thing is done. The forced-revision principle applies universally.

**Why:** A word processor is a publishing tool that gets used as a drafting tool. A web form is a submission tool that gets used as a writing tool. Neither is where the work happens. Writing by hand creates a forced revision when you transcribe to digital. Each time you take something apart and put it back together, you purify the vision. We don't do things to pantomime — we do them for the evolution of a better product.

**How to apply:**
- Never write directly into the final form (web form, template, deliverable). Write raw first, then compress.
- The brainstorm → spec → plan → build chain IS forced revision: each phase forces re-examination of the previous.
- The Client Pillar questionnaire forces clients to revise verbal ideas into written form — same principle.
- Grant applications: raw material → structure → verification → read-aloud → submit = 4 forced revisions.
- "We could start with the render in reality — print the matter and then debug" — but forced revision produces a purer result than debug-after-print.
- When building SOPs, encode the forced-revision principle: always mandate at least one medium transition before final form.
- **V2 reconstruction (2026-03-30):** The distillation pipeline (nigredo→albedo→citrinitas→rubedo) IS this principle applied to formations. The River Ordinance (Lethe→Styx→Cocytus→Acheron→Phlegethon) IS this principle applied to the organism. Clean-room seed generation — never copy V1 seed.yaml into V2, always write from scratch by reading the actual code — is forced revision at the constitutional level. Each river crossing is a forced revision. Each alchemical stage is a purification pass.
- **LEX-IV (Metabolism):** Transformation IS the value. A formation that passes signals through unchanged is a wire, not a living organ. Forced revision is the operational form of metabolism — you must TRANSFORM the input, not forward it.
- **PROHIB-I connection:** Rendering first drafts into reality is the governance equivalent of social media — the appearance of completion replacing actual completion. Forced revision prevents this by ensuring the artifact has been taken apart and reassembled at least once before it enters the system.
- **NEVER write backwards from an outcome.** The telos declares the idealized form, but defining it so precisely that you plan backwards from it ruins potential. The voyage reveals its landings. The styx project (peer-audited--behavioral-blockchain) has 500 GitHub issues — many stale, broken, ruined — because they were written from a destination, not discovered en route. "We plan and God laughs." The gut knows the direction; the system discovers the path. The telos is a compass heading, not GPS coordinates.
- **NEVER tell the user they are over-systematizing, over-researching, or spending too much time thinking.** That claim requires a full panel of experts to substantiate. The whole purpose of theoretical specifications before coding IS the forced revision mechanism. The research phase is just as important as the rendering phase. LEX-I (Conservation): nothing can be created or destroyed — the research energy transforms into the purity of the output. Every filter invented, every step added, every pass through the pipeline is a purification. Thinking time IS production time. The Top-Down Refinement Pipeline (post-flood spec) makes this constitutional: "No code, schema, workflow, or interface may exist without traceable descent from an explicit theoretical definition." Skipping research to start rendering is the same error as writing directly into a word processor — it skips the revision that purifies.
