`~` and `~/Workspace` are Apple's / tool conventions — "someone else's ideal as-estimated."
The user does NOT want to reorganize, re-nest, or force structure into these paths.
They should remain default, non-interacted locations for tools that expect them.

`~/source--all-ever` is the user's OWN system — designed from their ontology, not Apple's.
This is the clean entry point. The layers inside express the user's design philosophy using
Platonic/Aristotelian metaphysics as directory grammar:

- eidos--kosmos (intelligible world) → topos--noetos, methexis, chora
- hyle--kosmos (material world) → ousia, energeia, dynamis

`~/Workspace` receives PROJECTIONS (symlinks via methexis) from source--all-ever, so tools
find things where they expect them. But the structural truth lives in source--all-ever.

The recursive systems-within-systems design: each containment layer IS a system containing
systems. Not labels on folders — ontological constraints on what can exist at each level.

**Why:** User has spent excessive time organizing and fighting with ~/Workspace conventions
instead of building. The separation lets them design freely in source--all-ever while tools
remain happy with ~/Workspace's flat surface.

**How to apply:** Never try to restructure ~/Workspace itself. Never fight macOS/tool
conventions. Build in source--all-ever, project into ~/Workspace via symlinks. When the
user discusses workspace architecture, the vocabulary is metaphysical, not technical.
