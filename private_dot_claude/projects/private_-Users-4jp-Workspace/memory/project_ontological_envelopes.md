On 2026-04-03/04, a full-session deep dive produced the architecture for a "living container" — a directory hierarchy where every directory is a system with declared function, purpose, boundary, limits, inputs, outputs, and proofs.

**Key artifacts (committed to organvm-i-theoria/call-function--ontological/docs/):**
- `ontological-envelopes-diagrammatics.html` — 19 diagrams comparing macOS/Windows/Linux directory ontologies + the container architecture
- `container-architecture.md` — formal spec: axioms, schema, levels, naming grammar, coupling protocol, domain ontology, validation rules

**Core concepts developed:**
- Three OS directory systems analyzed as "ontological envelopes" (Editorial/Bureaucratic/Transparent)
- All three fail at home directory governance (the "ungoverned stratum" / dotfile seam)
- The container lives in "the void" — the region no OS claims
- Three irreducible operations: objectify (make primitives), couple (assemble interactions), contain (collect into envelopes)
- Curry-Howard-Lambek correspondence: logic/math/computation are one structure — system.yaml is simultaneously proposition, type, and specification
- Proof-carrying systems: every directory maintains proof of consistency, type-soundness, and execution
- CRITICAL PIVOT: emergence-first, not thesis-first. Don't pre-define domains/levels — let structure emerge from observed behavior, then formalize retroactively
- The ideal = the difference between all instances (not any single instance)
- Self-correcting seed function Q(here) → shape: placed at any location, reads environment, determines identity, self-corrects until stable
- 30 design questions (approach-agnostic) for building any such system

**Why:** Foundation for the user's `source--all-ever/` project — a self-describing, self-correcting, emergence-first container for natural/social/digital systems.

**How to apply:** This is ORGAN-I foundational theory. The architecture applies to any future system design work. The 30 questions are reusable for any project's initial design phase. The emergence-first protocol supersedes any top-down hierarchy planning.
