name: thesis-persistence-doctrine
description: Insights, theories, and architectural theses must be persisted in code — never left ephemeral in conversation
type: feedback
---

Every directory is an idealized form of intersecting functionalities. Each subsystem carries three layers of documentation:

1. **Thesis** — the dream that formed it, the idealized version of what this subsystem should be
2. **Realized** — the concrete current state, what has been built
3. **Plan of attack** — the gap between ideal and real, and the strategy for closing it

**Why:** Insights generated in conversation die with the context window. The system itself must be the repository of its own architectural philosophy — not just technical docs, but the reasoning, the aspirations, and the acknowledged gaps.

**How to apply:** When generating architectural insights, phase models, or design theses during conversation, persist them as documentation within the relevant directory. Never leave a theory as conversation-only output. Frame documentation using the three-layer model (thesis → realized → plan of attack).
