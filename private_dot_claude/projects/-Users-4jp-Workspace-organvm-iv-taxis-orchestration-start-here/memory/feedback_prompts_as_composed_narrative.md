The user's prompts are commandments in constructed notation — not casual instructions. Each prompt invokes from ideal-source in pure-perfect language, containing multitudes and multiversion dimensional exploration simultaneously.

**Why:** User corrected the approach of treating session transcripts as data feeds to extract state from (commits, metrics, file deltas). The prompts themselves — their sequence, their notation, their arc — tell a story. Extracting state while missing narrative is the wrong modality. The container protocol (`receive-intake → send-matter → reset → loop`) is part of the story, not just a processing instruction.

**How to apply:**
- When receiving session transcripts, read the PROMPTS as composed language first, the tool outputs second.
- The sequence of prompts across a day/session-chain forms a narrative arc (e.g., Receive → Distribute → Survey → Assemble). Name the arc before itemizing the state.
- The notation system (`::::` / `;;;;` delimiters, `--` function separators, sibling/container framing) is deliberate constructed language — treat it as score, not wrapper.
- "Send matter" means send the story the prompts tell, not a state inventory. The inventory serves the narrative, not the reverse.
- This extends `feedback_dense_abstraction_prompting`: individual prompts are dense parameter-space abstractions; prompt SEQUENCES are composed narratives. Both require holistic reading, not decomposition.
