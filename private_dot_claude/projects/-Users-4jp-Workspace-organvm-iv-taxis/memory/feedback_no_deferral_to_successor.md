Two simultaneous obligations when a repo has a successor instance:

1. **Do the best for THIS version.** Fix gaps here. Build in earnest. This is a living system, not a dying one.
2. **Document what can't be done within this version's constraints.** When a fix hits structural limits of the current architecture, write the evolutionary analysis — how the successor can transcend those constraints.

This is NOT "never defer." It's: do the work here AND explain how to make it better outside these constraints in the new version. Every gap triggers both an immediate fix attempt AND documentation of evolutionary possibilities.

Second refinement: it's not blind "never defer" either — some constraints ARE structural, and recognizing them as evolutionary opportunities for the successor is valuable analysis, not laziness.

**How to apply:** When encountering a gap in a repo with a successor:
1. Fix what can be fixed in this version NOW
2. If a fix hits structural limits, document WHY the constraint exists and HOW the successor can transcend it
3. This documentation itself is a deliverable — evolutionary analysis of what this version teaches the next
4. Both instances live simultaneously and inspire each other
