The organism absorbs CAPABILITIES, never implementations. This is the cardinal rule of the cocoon process.

**Why:** The predecessor's names (registry, dispatch, session) are engineering nouns from that iteration. The organism speaks in biological verbs derived from SEED axioms. Copying names or implementation details would import the predecessor's architectural compromises (isotope duplication, API coupling, guard clauses) into the new form. Purity requires re-derivation from first principles.

**How to apply:**
- Gate contract `sources` = what to study for understanding the capability
- Gate contract `identity` = derived from SEED grammar (mechanism + verb), NEVER from source naming
- When building a new function: derive it through SEED §II Procedure 1 from axioms. The ONLY thing that crosses from predecessor to organism is bare intent — "this capability must exist." No API shapes, no architectural patterns, no implementation suggestions, no "better ways" from the predecessor.
- Even well-intentioned structural suggestions from predecessors are INFECTIONS. "It had node discovery, maybe the organism should too" — that is the predecessor's ghost reaching through the glass. Block it. Derive fresh from axioms + present need.
- Never name a function or variable after its predecessor equivalent. `registry` → `skeletal--define`. `dispatch` → `circulatory--route`. The verb survives. The noun dissolves.
- This is alchemy (dissolve + reconstitute), not engineering (refactor). The middle location where past meets present transfers only the raw intention — the bare verb — and nothing else.
